## What This Changes — Build vs Import

`agent-installer` (cat-09) already handles install/remove/search from the repo.
`workflow-orchestrator` (cat-09) already handles general workflow orchestration.
`multi-agent-coordinator` (cat-09) handles the concurrent fan-out.
`pied-piper` (cat-09) handles SDLC agent team orchestration.

This reduces custom builds from 19 → 17 and eliminates building agent-manager entirely.

### ★ BUILD — 17 custom agents (genuinely don't exist anywhere)

```
ORCHESTRATION
  master-orchestrator    ← extends workflow-orchestrator with our pipeline routing
  issue-executor         ← GitHub issue → pipeline trigger, unique to this system

DISCOVERY & ANALYSIS
  project-analyst        ← new project stack detection, writes CLAUDE.md
  codebase-auditor       ← deep existing/legacy audit + AUDIT.md
  description-writer     ← writes .workflow/PROJECT.md from discovery
  tech-debt-mapper       ← categorises debt, estimates effort, creates issues

  architecture-challenger← devil's advocate, stress-tests every proposal

PLANNING

  task-decomposer        ← atomic tasks, dependencies, commit checkpoints

TESTING
  test-engineer          ← writes/updates tests, fully stack-aware, mandatory

DOCUMENTATION

COMMIT & CLOSE

## GitHub Issues — Universal Trigger

### Label System

```
WORKFLOW TYPE
workflow:feature       Full feature pipeline
workflow:bugfix        Bug fix + regression test + patch
workflow:refactor      Behavior-preserving refactor
workflow:modernize     Legacy migration step
workflow:audit         Full codebase analysis
workflow:brainstorm    Architecture debate → ADR
workflow:task          Small atomic task, no branch

PROJECT CONTEXT
project:new            Fresh project
project:existing       Active codebase
project:legacy         Legacy/old codebase

PRIORITY
priority:critical      Execute immediately
priority:high
priority:normal
priority:low

SOURCE
source:manual          Human created
source:external        External system (Sentry, CI, monitoring, etc.)
source:audit           Created by codebase-auditor
source:architecture    Created from ADR
```

### Required Issue Body Format

```markdown
## Summary
[What needs to be done — 1-2 sentences]

## Context
[Why this matters, background]

## Acceptance Criteria
- [ ] Criterion 1
- [ ] Criterion 2

## Technical Notes
[Stack details, constraints, related issues]

## Definition of Done
- [ ] Code written and working
- [ ] Tests written and passing
- [ ] Coverage threshold met
- [ ] Review passed (code + security + performance)
- [ ] Docs updated (CHANGELOG + API + inline)
- [ ] Issue closed with summary comment

<!-- WORKFLOW META -->
workflow_type: feature|bugfix|refactor|task
project_type: new|existing|legacy
estimated_complexity: small|medium|large
related_issues: #123, #456
```

## All Pipelines

### Pipeline 1 — New Project
```
/workflow:discover DISCOVERY.md
         ↓
  project-analyst ★
  ├── Reads any format: notes, brief, spec, README
  ├── Detects stack → populates CLAUDE.md
  ├── Detects provider (GitHub/Linear/ClickUp/Jira)
  └── Recommends specific agents to install
         ↓
  description-writer ★
  └── Writes .workflow/PROJECT.md
         ↓
  /agents:install [recommended list]
         ↓
  → Pipeline 4 (Feature) for first issue
```

---

### Pipeline 2 — Existing Project Onboarding
```
/workflow:audit
         ↓
  codebase-auditor ★  +  security-auditor ✅ (parallel)
  ├── Architecture (patterns, boundaries, violations)
  ├── Dependency audit (outdated, deprecated, vulnerable)
  ├── Code quality (duplication, complexity, dead code)
  ├── Security (secrets, injection, auth patterns)
  ├── Test coverage gaps
  └── Writes .workflow/AUDIT.md
         ↓
  compliance-auditor ✅ (if regulated project)
         ↓
  tech-debt-mapper ★
  ├── 🔴 Critical — fix now (security/data risk)
  ├── 🟠 High — this sprint
  ├── 🟡 Normal — address soon
  ├── 🟢 Low — nice to have
  ├── Estimates effort per item
  └── Writes .workflow/TECH-DEBT.md
         ↓
  feature-planner ★
  └── Creates GitHub issues for all debt items
      labels: workflow:refactor, priority:X, source:audit
```

---

### Pipeline 3 — Legacy Project Modernisation
```
/workflow:audit --legacy
         ↓
  codebase-auditor ★ (legacy mode — deeper scan)
  + legacy-modernizer ✅ (modernisation scope assessment)
  + dependency-manager ✅ (EOL/deprecated map)
         ↓
  /arch:brainstorm "safest migration path for this system"
  ├── architecture-lead ★ (3 phased migration approaches)
  ├── architecture-challenger ★ (stress-tests each)
  ├── knowledge-synthesizer ✅ (structures debate)
  └── [You guide direction]
         ↓
  architecture-recorder ★
  └── ADR + phased GitHub issues
         ↓
  migration-planner ★
  ├── Phase 1: Critical security (workflow:bugfix, priority:critical)
  ├── Phase 2: Dependency updates (workflow:task)
  ├── Phase 3: Pattern modernisation (workflow:refactor)
  └── Phase 4: Architecture improvements (workflow:feature)
         ↓
  Execute phase by phase: /workflow:issue <n>
```

## Issue Creation Decision Flow

After any plan, architecture decision, or bug fix — you choose what happens next.
The system never assumes. It always presents the decision explicitly.

### The Three Moments This Appears

**1. After `/workflow:plan "description"` or `/workflow:feature`**
**2. After `/arch:decide` (ADR written)**
**3. After `/workflow:fix` (bug fixed, patch written)**

### Decision Prompt Format

After completing any of these, the system presents:

```
✅ [Plan / ADR / Fix] complete.

What would you like to do next?

  [A] Create GitHub issue + implement now
      → Creates issue with full context, then immediately executes it
      → Issue has complete record in GitHub history

  [B] Create GitHub issue only (implement later)
      → Creates issue, adds to backlog
      → Execute anytime with: /workflow:issue <n>

  [C] Implement directly (no issue)
      → Runs pipeline immediately, no GitHub issue created
      → Use for quick tasks where tracking overhead isn't worth it

  [D] Save locally only (plan / patch / ADR stays in .workflow/)
      → Nothing pushed, nothing executed
      → Review and decide manually later

Type A, B, C, or D:
```

### How Each Option Works

#### Option A — Create issue + implement now
```
issue-creator ★ builds issue body from:
├── Plan: tasks, acceptance criteria, complexity estimate
├── ADR: decision, implementation phases, risks  
├── Fix: problem description, solution summary, regression test
└── Applies correct labels automatically

GitHub MCP creates the issue
         ↓
issue-executor immediately picks up the new issue number
         ↓
Pipeline runs as normal
         ↓
issue-closer closes it with full summary at the end
```

#### Option B — Create issue only
```
Same issue creation as Option A
         ↓
Issue sits in GitHub backlog with all context
         ↓
Execute anytime: /workflow:issue <n>
         ↓
Useful when: building a backlog, prioritising later,
             sprint planning, or external systems will pick it up
```

#### Option C — Implement directly
```
Pipeline runs immediately without creating an issue
         ↓
Useful when: hotfix needed right now,
             exploration/spike work,
             local-only change not tracked in GitHub
         ↓
Note: no GitHub record is created.
      A .workflow/patches/ entry is still written for bug fixes.
```

#### Option D — Save locally only
```
Plan → .workflow/PLAN.md or features/
ADR  → .workflow/ADRs/
Fix  → .workflow/patches/
         ↓
Nothing pushed. Nothing executed.
Review later, decide what to do.
         ↓
Useful when: brainstorming, not ready to commit to the work,
             want to review the plan before it becomes a real issue
```

## Plan → Issues Strategy

When a big plan is created (audit findings, feature decomposition, migration
phases, architecture implementation), you have a choice about how to handle
all the tasks. Here is the decision framework and recommended approach.

### The Core Question

```
Big plan created with N tasks / phases
         ↓
Do you push ALL of it to GitHub issues at once?
Or keep it local and create issues one at a time?
```

### The Answer: It Depends on Plan Type

| Plan Type | Recommended Strategy | Why |
|-----------|---------------------|-----|
| Audit findings (TECH-DEBT.md) | **Push ALL to GitHub** | These are your backlog. You want them visible, labelled, prioritised, and assigned. External team members need to see them. Sprint planning needs them. |
| Feature task breakdown | **Keep local, push parent issue only** | Individual tasks inside a feature are implementation details. GitHub doesn't need a separate issue for "write the unit test for UserService". The parent feature issue tracks it all. |
| Migration phases (legacy) | **Push each PHASE as an issue** | Phases are meaningful units of work, reviewable and executable independently. Tasks within a phase stay local. |
| Architecture ADR implementation | **Push one issue per ADR** | The ADR is the spec. One issue = one implementation. Breaking it into sub-issues is overhead unless it spans multiple weeks. |
| Bug fix patch | **Push one issue per fix** | Each fix is independent, testable, and closeable on its own. |

### Bulk Issue Creation — `/workflow:push-plan`

When you want to push an entire plan to GitHub at once:

```
/workflow:push-plan [--audit | --features | --phases | --all]
         ↓
  issue-creator ★ runs in batch mode
  ├── Reads source (.workflow/TECH-DEBT.md or PLAN.md or features/)
  ├── Creates one issue per item
  ├── Applies labels automatically
  ├── Links related issues (parent/child via "related_issues" field)
  ├── Maintains order as GitHub milestone or project board column
  └── Reports: "Created 12 issues: #101, #102, #103 ..."
```

### Recommended Approach per Scenario

#### Scenario A: You just ran /workflow:audit on an existing project

### Local vs GitHub — The Rule of Thumb

```
Push to GitHub when:
  ✓ It's a meaningful, reviewable, closeable unit of work
  ✓ Other people need to see it exists
  ✓ It needs to be tracked in sprints / milestones
  ✓ It can be executed independently from other tasks
  ✓ External systems might pick it up

Keep local when:
  ✓ It's an implementation detail of a larger tracked item
  ✓ It will be completed as part of a parent issue execution
  ✓ It's exploration / spike work, outcome uncertain
  ✓ It's a sub-task that doesn't need individual visibility
```

## Workflow Self-Improvement — `/workflow:improve`

This command is for improving the workflow system itself.
Not your project code. The workflow. The agents. The commands. The rules.

When something feels wrong, slow, missing, or broken in how the system
works — this is where you fix it.

### What It Does

```

### What Gets Improved Typically

**Agents** — instructions too vague, wrong tool permissions, missing edge cases  
**Commands** — unclear triggers, missing flags, wrong agent routing

**Testing rules** — coverage thresholds too strict/loose for project type  
**Doc triggers** — wrong events triggering doc updates, missing events  
**Issue templates** — fields nobody fills in, missing fields agents need
